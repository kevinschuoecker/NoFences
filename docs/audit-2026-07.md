# Code audit, July 2026 (solo alpha)

Scope: FenceWindow lifecycle, persistence timing, threading, Win32 layer.
Policy: only P0/P1 fixed; P2 documented and deferred.

## Fixed in this audit

| ID | Prio | Finding | Fix |
|---|---|---|---|
| A1 | P0/P1 | Debounced saves could fire up to 4s after a fence was deleted (or after a layout import) and re-create the just-deleted metadata folder - deleted fences resurrected on next start | `SuppressPersistence` flag set by both removal paths; `Save()` no-ops when set |
| A2 | P1 | `Screen.AllScreens` is cached once by .NET Framework; the off-screen rescue consulted a stale monitor layout exactly when displays changed | New `MonitorUtil` (Win32 `MonitorFromRect`/`GetMonitorInfo`, always live) |
| A3 | P1 | `animTimer`, `highlightTimer`, `widgetTimer` and the CPU `PerformanceCounter` were never disposed - closed fences kept ticking/polling | Disposed in `FormClosed`; pending position/note saves flushed first |
| A4 | P1 | `ThrottledExecution` was `async void` (exceptions escaped the caller) and could invoke the action multiple times per wait loop; pending work was lost on close | Rewritten as a WinForms-timer trailing debounce with `Flush()`/`Cancel()`; flushed on close |

## Documented, deferred (P2)

- `SnapToNeighbors` uses `Screen.FromControl` work areas (same stale-cache
  family as A2); impact is a slightly-off snap right after a display change.
  Revisit if reported.
- `ShellContextMenu` `NativeWindow` handle is never destroyed (one per
  fence); tracked anyway for replacement per ADR/roadmap.
- Shell icon cache is unbounded; grows only with extensive portal browsing.
- Sample widgets use `WebClient` without an explicit timeout (default 100s);
  a hung request delays that widget's refresh but never blocks the UI.
- Hit-testing during paint (inherited design) - scheduled for the year-2
  renderer extraction, not worth destabilizing the alpha now.

## Bug-class takeaways (guidelines)

1. **Deferred work must check liveness**: anything debounced/queued that
   touches persistent state needs a suppression/cancellation path for the
   "owner was deleted meanwhile" case.
2. **Every `Timer`/`IDisposable` created by a window is disposed in
   `FormClosed`** - WinForms timers do not stop by themselves.
3. **No `async void` outside event handlers.**
4. **Prefer live Win32 queries over framework caches** for display topology
   (`Screen.AllScreens`, `Screen.PrimaryScreen` after display changes).
